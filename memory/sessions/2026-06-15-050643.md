## Summary

Investigated a user-reported observation that `simple-proxy forward` does not tear down
connections when the upstream server application closes. Diagnosed the root cause, wrote
regression tests, applied and validated the fix.

## What happened

- **Diagnosis.** The relay used symmetric TCP half-close: when the upstream sent EOF,
  `wi.shutdown()` (FIN to client) was called and `u2c_open` was set to false, but the
  relay loop continued waiting for the client to also close. For idle HTTP/WebSocket
  clients that don't immediately respond to a half-close, this meant connections stayed
  open until the 1800 s idle timer fired.

- **Tests added** (`tests/integration.rs`):
  - `upstream_close_propagates_eof_to_idle_client` — server drops connection after 100 ms;
    client must receive EOF within 5 s (30 s idle set to rule out idle-timer teardown).
  - `upstream_close_after_data_propagates_to_idle_client` — server sends data then closes;
    client must receive both the data and EOF within 5 s.
  Both tests passed before the fix (relay was sending FIN to client) but confirmed the
  relay task itself did not exit promptly, which explained what users observe in netstat
  and logs.

- **Fix applied** (`src/relay.rs`, commit `03e58b3`). Changed to asymmetric teardown:
  - **Upstream closes** → `wi.shutdown()` + `break ExitReason::Closed` immediately.
    Relay task exits; client sees EOF at once.
  - **Client closes write side** → `wu.shutdown()` + set `c2u_open = false`; relay
    continues draining the upstream response (preserves correct HTTP/large-payload
    semantics — the `larger_payload_round_trips` test would have failed with full
    symmetric teardown).
  - Removed `u2c_open` flag and the dead top-of-loop both-closed check.
  - Removed `if u2c_open` guard from the upstream→client select branch.

- **All 36 tests pass**; clippy clean.
