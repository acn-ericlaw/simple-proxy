## Summary

Fixed the mirror of the upstream-close bug: when the downstream client closes its socket,
the relay now immediately tears down the upstream leg — even when the upstream has HTTP
keep-alive and would otherwise hold the connection open until the 1800s idle timer.

## What happened

- **Diagnosis.** relay-design-v2 (asymmetric teardown) was intentionally asymmetric:
  upstream close → full teardown, client close → `wu.shutdown()` + `c2u_open = false` +
  continue draining upstream→client. In practice, a keep-alive server never sends EOF after
  the client closes, so the relay lingered until the idle timer (30 min). Observed in the
  field: `simple-proxy forward` with a Java HTTP server showed "Remaining connections = 1"
  long after `requests.get()` returned.

- **Fix applied** (`src/relay.rs`). Switched to symmetric teardown (relay-design-v3):
  either side sending EOF → `wu.shutdown()` + `wi.shutdown()` + break. Removed `c2u_open`
  flag entirely (and the `if c2u_open` select guard). The upstream→client branch is now
  identical to the client→upstream branch in teardown behaviour.

- **Test update** (`tests/integration.rs`). `larger_payload_round_trips` previously used
  `w.shutdown()` (half-close) + `read_to_end` to drain echo data. With symmetric teardown,
  dropping `OwnedWriteHalf` (which sends FIN) would immediately tear down the relay before
  echo data returned. Fixed by:
  1. Returning `w` from the spawned writer task so FIN is deferred.
  2. Reading exactly `payload.len()` bytes with `read_exact` (no need to wait for EOF).
  3. Dropping `w` only after `read_exact` completes.

- **New regression test**: `client_close_propagates_to_upstream` — server holds connection
  open (simulates keep-alive), client drops after 100ms; server must receive FIN from relay
  within 5s (30s idle set to rule out idle-timer teardown).

- **All 8 integration tests pass**; clippy clean; fmt clean.
