## Summary

Follow-up to the keep-alive teardown fix (relay-design-v3). A user reported that with
Python `requests`, one TCP connection stays open after `requests.get()` returns (also seen
with browsers). Investigated, confirmed it is **client-side HTTP keep-alive + CPython GC**,
not a proxy bug, and documented it as a README FAQ so users stop filing it as a defect.

## What happened

- **Observation (from proxy log).** Each new `requests.get()` opened a new session AND
  closed the *previous* session; the final connection closed only when Python exited.

- **Reproduced** with `uv` (system python3 is locked — no pip). Created a throwaway venv
  (`requests==2.34.2`, `urllib3==2.7.0`), a small HTTP/1.1 keep-alive server, and an `lsof`
  probe around `requests.get()` calls. Findings:
  - After `get()` *returns*, the socket is still `ESTABLISHED` (keep-alive; neither end
    closed it).
  - Rebinding `r = requests.get(...)` closes the old socket (old `Response` GC'd) and opens
    a new one.
  - `del r; gc.collect()` → 0 connections. The socket closes only when the Python object
    holding it (`r.raw` → urllib3 connection) is garbage-collected.
  - Matches the field log exactly. (`lsof -p <pid> -i` is OR'd, so each loopback connection
    showed 2 rows = the two endpoints; one real connection.)

- **Root cause.** Module-level `requests.get()` keeps the socket alive via the returned
  `Response`; it closes on GC (reassignment / `del` / interpreter exit), on the server's
  keep-alive timeout, or on `simple-proxy`'s `idle_timeout_secs`. The proxy correctly holds
  an L4 connection open as long as both ends do (relay-design-v3 already tears down promptly
  when EITHER side actually closes). So this is expected, not a leak.

- **Documentation.** Added a "FAQ: a connection stays open after my HTTP request finished"
  section to `README.md` (before "Running as a service"): explains keep-alive, the Python
  `requests`/`urllib3` GC behavior with a code example, the eager-close workarounds
  (`with Session()` / `r.close()`), equivalent browser connection-pool behavior, and the
  three ways a keep-alive connection eventually closes.

- No code change; no new tests. Reproduction artifacts cleaned up.
