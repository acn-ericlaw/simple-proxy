## Summary

Investigated and fixed a **failing GitHub Actions CI** (red on every recent run, including
commits that touched only `memory/`). Root cause was toolchain drift, not the code changes:
the workflow used a floating `dtolnay/rust-toolchain@stable` together with
`RUSTFLAGS: -D warnings`, and stable rolled forward 1.95 → 1.96, which promoted the clippy
lint `items_after_test_module` to warn-by-default — turned into a hard error by `-D warnings`.
Fixed the lint and **pinned the toolchain** so this class of breakage can't recur silently.

## What happened

### 1. Diagnosis
- `gh run list` — CI `failure` on the last four pushes (18–34s each → failing early).
- `gh run view --log` — exit code 101 in the clippy step:
  `error: items after a test module --> src/proxy.rs:250`,
  `note: -D clippy::items-after-test-module implied by -D warnings`. The lint doc URL
  referenced `rust-1.96.0` while local builds are `1.95.0` → confirmed the floating-stable drift.

### 2. Code fix (`src/proxy.rs`)
Moved the `ConnGuard` struct + its two `impl`s **above** the `#[cfg(test)] mod tests` block
so the test module is the last item in the file (what the lint requires). Pure reordering —
no behavior change. Checked the other seven `mod tests` blocks (event-bus lib, shutdown, log,
config, discovery, allowlist, cli) — `proxy.rs` was the only one with trailing items.

### 3. Toolchain pin (maintainer-chosen hardening)
Offered three options (pin / fix-reactively / drop `-D warnings`); maintainer chose **pin**.
- Verified that `dtolnay/rust-toolchain` does **not** read `rust-toolchain.toml` (its toolchain
  is selected by the `@rev`), so the pin can't be honored by that action.
- Added `rust-toolchain.toml` (`channel = "1.95.0"`, `components = ["rustfmt", "clippy"]`) as the
  single source of truth for local + CI, with a comment explaining the drift it prevents.
- Edited `.github/workflows/ci.yml`: replaced the `dtolnay/rust-toolchain@stable` step with
  `run: rustup show && rustc --version`, which makes rustup auto-install the pinned channel +
  its components from the toml. `RUSTFLAGS: -D warnings` and the three gate steps are unchanged.

### 4. Verification (local, under the pinned toolchain)
`rustup show` confirms `active because: overridden by rust-toolchain.toml` → `rustc 1.95.0`.
- `cargo fmt --all --check` → clean
- `RUSTFLAGS="-D warnings" cargo clippy --workspace --all-targets` → clean
- `cargo test --workspace` → **48 pass** (7 event-bus + 33 proxy unit + 8 integration)

### 5. Experiment: 1.96.0, then bumped the pin (maintainer follow-up)
Maintainer asked to experiment on 1.96.0 and consider pinning to the latest. Installed
`1.96.0` (current latest stable, released 2026-05-28) with rustfmt+clippy and ran all gates
under it: `cargo +1.96.0 fmt --all --check` clean, `RUSTFLAGS="-D warnings" cargo +1.96.0
clippy --workspace --all-targets` clean, `cargo +1.96.0 test --workspace` 48/48. The
`ConnGuard` reorder was the *only* thing 1.96 objected to — no other new clippy lints, and
rustfmt 1.96 wants no reformatting. So **bumped the pin from 1.95.0 → 1.96.0** in
`rust-toolchain.toml` (better: current, and matches what CI's floating stable was already
using). Re-verified through the toml override: `rustup show` → `overridden ... 1.96.0`, all
three gates green. (Note: local `stable` was stale at 1.95.0 — the real latest stable is 1.96.0.)
